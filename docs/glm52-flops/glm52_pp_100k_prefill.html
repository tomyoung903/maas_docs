GLM-5.2 PP in a 100K Prefill

Pipeline parallelism splits the layer stack across GPU stages. It does not split the token axis like context parallelism, and it does not split each matrix multiply the same way tensor parallelism does. In a long prefill, PP mainly changes which layers live on which GPUs and how chunked prefill microbatches flow through those stages.

PP is therefore not the same kind of speedup as DSA context parallelism. CP reduces the long-attention work assigned to one rank. PP preserves the layer dependency and overlaps different microbatches across stages after the pipeline is filled.

PP Math

Let the model have L transformer layers and S pipeline stages. GLM-5.2 has 78 transformer layers in the SGLang runtime path used here, and the experiment used four PP stages.

Layer ownership
S=4 pipeline stages
L=78 runtime transformer layers
base=floor(L / S) = 19
remainder=L mod S = 2
PP0, PP1own19 layers each
PP2, PP3own20 layers each extra layers go to the last remainder stages

For a microbatch m, stage 0 computes its layers first, sends the hidden states to stage 1, and so on. Tensor parallelism still happens inside a stage: in this run, every PP stage had two TP ranks.

Microbatch flow
Hm,0→Stage0 layers [0, 19)
Hm,1→Stage1 layers [19, 38)
Hm,2→Stage2 layers [38, 58)
Hm,3→Stage3 layers [58, 78) → logits

If each stage takes about tau seconds per microbatch and there are M microbatches, the ideal balanced pipeline time is not M S tau. After the initial bubble, different stages work on different microbatches.

Idealized timing
Tserial≈M × S × tau
Tpipeline≈(M + S - 1) × tau
bubble=(S - 1) × tau before the first fully processed microbatch exits

100K Chunking

A single 100K-token request is too large to run as one prefill batch under the configured 8,000-token prefill cap. The scheduler repeatedly admits a chunk, updates the request state, then admits the next chunk until the prompt is consumed.

Configured chunk math
T=100,000 prompt tokens
C=8,000 max prefill tokens per scheduler chunk
Nchunks=ceil(T / C) = 13
observed PP0 chunks=12 × 8,000 + 4,032 page-aligned tail in the log

The tail is 4,032 rather than exactly 4,000 because this run used a 64-token page size and the scheduler aligns chunk boundaries to pages. The server log for each hot 100K single request shows PP0 scheduling 13 prefill batches with #cached-token: 0.

SGLang Scheduler

The implementation matters because PP is not a separate external queue. The PP event loop repeatedly calls the normal scheduler for each PP loop slot, then sends request metadata and hidden-state proxy tensors between neighboring stages.

For a single long request, chunks of that one request can fill the pipeline after the first few steps. For multiple long requests, the scheduler has more pending chunks, but the measured completions show that the requests still finish as a staircase. Four concurrent 100K prompts returned at about 5.16, 9.47, 13.46, and 17.47 seconds.

Interpretation

PP improves feasibility and can improve throughput when there are enough microbatches, but it does not erase layer order. Every token chunk still passes through PP0, then PP1, then PP2, then PP3. The first output waits for the pipeline fill bubble, and the last output waits for the pipeline drain.

This is why PP should be mentally separated from DSA prefill context parallelism. CP changes how the long attention scan is divided over the token axis. PP changes how the layer stack is divided over GPUs. In GLM-5.2 long prefill, those are different bottlenecks.

An attempted no-PP TP8 baseline on the same node was excluded from timing: the TP8 server reached health only after disabling prefill CUDA graph, then crashed during the cold 16K warmup before producing usable 100K data. The successful PP4 data above is therefore the experiment evidence used for this page.
