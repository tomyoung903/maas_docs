Why GLM-5.2 DSA Prefill CP Is Faster

Context parallelism for GLM-5.2 long prefill is best understood as a query-token-axis decomposition of the DSA prefill path. It can be much faster than plain TP8 because the long-context bottleneck is the replicated DSA selector and sparse-attention plumbing, not a single ideal dense GEMM.

Short Answer

With plain TP8, each attention TP rank still sees the full prefill query chunk for the DSA selector path. The indexer modules are replicated, so a large part of the selector work is repeated across TP ranks. With CP8, those same ranks are relabeled into context-parallel ranks and each rank handles about one eighth of the query rows. That directly shortens the per-rank critical path for the long-context DSA selector and sparse-attention setup.

CP8 also means attention TP size becomes one in the measured single-node setup. The replicated selector is divided by query rows, while ordinary head-sharded attention work is mostly traded from head sharding to token sharding. The speedup is therefore not 8x: the run still includes projection work, MoE work, selected-key attention, local top-k compute, top-k/result metadata movement, KV/index-key all-gather or rerange, synchronization, scheduling, and kernel overhead.

This page builds on the DSA + MLA math page. That page defines the MLA latent cache, the DSA selector, and the selected-key sparse MLA attention. Here the question is wall-clock prefill decomposition.

Bottleneck Math

For a long prompt, SGLang does chunked prefill. In the measured setup, a full chunk has Tc = 8,192 query tokens, P = 8, and DSA keeps K = 2048 selected positions per query after the short-context warmup.

The key bottleneck is that the DSA selector has to evaluate current query rows against a long causal candidate history before it can return the top selected positions. That candidate table has natural shape query rows by candidate context. TP head sharding is good for per-head attention math, but it does not automatically remove a replicated selector over all query rows.

At 100K tokens, the observed server-side speedup of about 2.542x implies that roughly 69.3% of the baseline request time was effectively on the CP-divided path. The remaining 30.7% includes work that CP does not divide perfectly or that CP adds back through communication and orchestration.

Practical Reading

CP8 is worth using when long-prefill DSA selector and sparse-attention setup dominate the request. It is less useful for short prompts, decode-heavy traffic, or workloads where memory pressure and communication overhead erase the query-row split benefit.

For the current H200 setup, CP8 should be treated as a Hopper long-prefill optimization. It trades memory and communication for lower TTFT on long prompts. Leave it off for Blackwell paths unless the DSA-CP FP8 RoPE kernels and validation for that platform are in place.
