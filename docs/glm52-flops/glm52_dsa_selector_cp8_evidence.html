GLM-5.2 DSA Selector CP8 Evidence

This page records a direct H200 experiment for the DSA selector path during a 100K-token GLM-5.2 prefill. The only serving-mode difference is plain TP8 versus TP8 plus --attn-cp-size 8 DSA prefill context parallelism.

The important point is that CP8 cuts the per-rank selector query rows by 8x, but the measured selector critical path improves by about 2.64x, not 8x. The remaining time is fixed overhead, selector kernel efficiency, memory traffic, synchronization, and the rest of the model prefill path.

Caveats

  • The selector timing is for layer 0 only. The operation shape is representative of the selector path, but this page does not claim a full all-layer selector sum for the 100K prompt.
  • The hook synchronizes CUDA around Indexer.forward_cuda, so absolute end-to-end latency is perturbed. The comparison is still useful because both modes use the same hook and same prompt size.
  • The first measured request includes some residual cold-start effects. The headline numbers use requests 2 and 3.
  • The experiment used two H200 nodes, one per mode, so tiny node-to-node differences are possible. The row-count evidence is independent of that.
