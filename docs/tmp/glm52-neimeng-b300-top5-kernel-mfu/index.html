Answer First

Lowest meaningful MFU: sparse MLA attention, 30.84%. For this exact 16,384-token kernel shape, roughly 30% is structurally understandable. It is not a universal lower bound and it is not proof that further optimization is impossible.

The result is constrained by one resident CTA per SM, 220.672 KB shared memory per block, 128 registers per thread, 24.96% active-warps occupancy, and 66.69% long-scoreboard stall share. At the same time, only 8.08% of peak DRAM bandwidth is used and L2 hit rate is 94.37%. This is an irregular latency-and-dependency problem, not an HBM saturation problem.

Why Sparse Attention Lands Near 30%

This is the strongest answer to the original question. The analytical numerator, launch resources, and hardware counters tell the same story.

Numerator reconstruction

Npairs = 2,048 × 2,049 / 2 + 2,048 × (16,384 - 2,048) = 31,458,304
Wattention MMA = 31,458,304 pairs × 10 layers × 64 heads × 2 × (2 × 512 + 64) = 43.810 TFLOP
MFUattention = 43.810 / (0.031566 s × 4,500 TFLOP/s) = 30.84%

Another 0.1007 TFLOP of score, masking, and softmax-like scalar work is tracked separately. Converting that work to FP8 Tensor-Core equivalents would manufacture precision that the operation does not have.

Counter evidence

24.96%

HBM is not saturated
8.08%

Peak DRAM throughput, approximately 620.5 GB/s during the sampled launch.

Most traffic hits L2
94.37%

L2 sector hit rate, with 63.43% L1 sector hit rate. Sparse dependency latency remains despite low HBM demand.

There is enough grid work
110.7

Waves per SM from a 16,384-block grid. The problem is per-SM resident state and dependency hiding, not a grid too small to cover 148 SMs.

What “inevitable” means here. With the current 512-thread, 220.672-KB, 128-register kernel, two CTAs would require no more than 116.736 KB shared memory and 64 registers per thread. That is approximately a halving of both resources, not a launch flag. A materially higher MFU therefore requires kernel/data-layout redesign, less sparse latency, or a different algorithm. It is not available from a small scheduler tweak.

Occupancy Is a Constraint, Not the Score

All three tensor families run one persistent CTA per SM, yet their normalized MFU spans 30.84% to 76.98%. DeepGEMM is the counterexample that prevents a simplistic “low occupancy means low utilization” conclusion.

  • One 384-thread CTA consumes 219.136 KB and 168 registers per thread.
  • Gate/up is 38.21%; down is 32.07%.
  • The 128 by 128 by 128 tile has exact K/N divisibility, but per-expert token counts create M-tail and load-imbalance loss.
  • It is the largest time target even though attention has the lower MFU.

DeepGEMM

  • The sampled q_b launch has only 12.44% active warps but reaches 93.21% SM SOL.
  • Its persistent UMMA/TMA pipeline keeps hardware busy with few issuing warps.
  • This dominant compiled-name family reaches 76.98% useful normalized MFU, so it is not the first optimization target.
  • The legacy NCU pipe-TC percentage is not used: it conflicts with both SM SOL and model-FLOP accounting on these Blackwell kernels.

Why Gather and RoPE Have No MFU Number

A dense FP8 denominator answers “how much Tensor-Core matrix throughput did this kernel realize?” It does not answer whether an index gather or elementwise quantization kernel is efficient. Assigning either a 0% MFU would be numerically true only in a trivial sense and operationally misleading.

FamilyLaunchesNsys msObserved geometryCorrect efficiency lens
gatherTopK 30 4.950 20 launches: grid 16,384, block 256; 10 launches: grid 16,384, block 32 Memory transactions, branch/comparison issue, and TopK algorithm cost
RopeQuantizeKernel 10 4.108 Grid 512 × 585, block 4 × 32, 43 registers/thread Bytes moved, conversion throughput, and elementwise instruction rate
Both attempted full counter captures failed dependency-sensitive kernel replay and exported zero performance counters. Their Nsys timings and launch geometry remain valid; no bandwidth or issue-efficiency claim is made from the failed reports.
