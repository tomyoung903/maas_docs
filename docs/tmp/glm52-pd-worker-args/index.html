Summary

Short answer: --max-prefill-tokens 8192 on the decode worker is not the normal decode scheduling limiter. Decode mode does not schedule ordinary prefill batches through PrefillAdder; it creates prebuilt decode batches from already-transferred KV state and caps admission by request slots.

Caveat: the value is still carried through ServerArgs and worker info, so it can affect generic logging or buffer helper code. It is just not the knob that limits decode-side work in this PD path.

Decode uses --tp-size 8 and --dp-size 8 because DP attention reinterprets the TP world as attention-DP lanes. With Tglobal = 8 and Dattn = 8, the inner attention tensor-parallel size is:

Tattn = Tglobal / Dattn = 8 / 8 = 1

Practically, the pod runs eight scheduler/model ranks on eight GPUs; each rank is one attention-DP lane, and the controller dispatches across those lanes.

Prefill uses --max-prefill-tokens 16384, --chunked-prefill-size 16384, --pp-size 8, and --max-running-requests 48. Because pp_max_micro_batch_size is unset, the scheduler computes:

Bpp-micro = max(48 / 8, 1) = 6 requests

That request-count microbatch cap is separate from the token cap. The token cap is handled by prefill admission and chunking through PrefillAdder.

1. Decode --max-prefill-tokens 8192

In ordinary generation mode, max_prefill_tokens feeds the prefill scheduler. The scheduler creates a PrefillAdder with self.max_prefill_tokens and chunked_prefill_size. PrefillAdder then stops adding additional requests once the prefill input-token budget is consumed.

Pbudget = configured max_prefill_tokens, refined by chunking and available KV memory

Decode disaggregation is different. New requests are placed into the decode preallocation queue, not the ordinary waiting queue used by the prefill adder. The decode scheduler later constructs a fake completed prefill batch with prepare_for_prebuilt() and runs decode. 

_add_request_to_queue(...):
  NULL    -> waiting_queue
  PREFILL -> disagg_prefill_bootstrap_queue
  DECODE  -> disagg_decode_prealloc_queue

The decode prebuilt path caps new admissions by request slots: batch_size = min(req_to_token_pool.size, max_running_requests). It does not pass through PrefillAdder(max_prefill_tokens, ...).

So the practical answer is: for decode worker scheduling and TTFT-style prefill batching, --max-prefill-tokens 8192 is not the operative limiter. The meaningful decode knobs in this chart are max_running_requests, decode CUDA graph batch size, speculative decoding settings, available KV/cache slots, and DP-attention lane count.

It is still not a pure parser no-op. TPModelWorker.get_worker_info() returns server_args.max_prefill_tokens to the scheduler, and generic helpers such as eager buffer sizing can read related prefill ceilings. That is why the careful statement is "not a decode scheduling limiter," not "impossible to observe anywhere."

2. Decode tp-size=8 and dp-size=8

In this codebase, DP attention uses the TP process group as the physical eight-rank worker group, then derives attention-DP and attention-TP from it. The key function computes:

Dattn = dp_size when enable_dp_attention is true
Tattn = tp_size / Dattn / Cattn

Your decode chart has tp_size=8, dp_size=8, and attn_cp_size=1. Therefore:

Tattn = 8 / 8 / 1 = 1

This is not "64 GPUs." The distributed environment still initializes a world of tp_size * pp_size. For decode, pp_size=1, so the pod uses eight ranks on eight GPUs. DP attention makes each rank an attention-DP lane, and model/expert communication then gathers and scatters hidden states across the shared TP group where needed.

Quantity Value in decode worker Meaning
Tglobal 8 Eight physical scheduler/model ranks, one per GPU.
Dattn 8 Eight attention-DP lanes for independent request partitions.
Tattn 1 No inner attention tensor sharding inside each DP lane.
Per-lane request cap About 16 max_running_requests 128 / dp_size 8, subject to KV capacity.

3. Prefill max-prefill-tokens 16384 and pp-size 8

The prefill workers use pipeline parallelism, not decode DP attention: tpSize=1, dpSize=1, ppSize=8. Each pod still uses eight GPUs, but those GPUs are pipeline stages rather than attention-DP lanes.

Token chunk cap
16,384
PP stages
8
Computed microbatch cap
6 requests

The code has two separate limits:

  • Token budget: max_prefill_tokens and chunked_prefill_size constrain how many prefill input tokens are scheduled into an extend batch.
  • PP request microbatch cap: pp_max_micro_batch_size constrains how many requests can sit in one PP microbatch slot.

Because the chart does not set --pp-max-micro-batch-size, scheduler initialization fills it in:

Bpp-micro = max(max_running_requests / pp_size, 1)
Bpp-micro = max(48 / 8, 1) = 6

Then prefill admission calls get_num_allocatable_reqs(running_bs), which subtracts the current PP microbatch request count from pp_max_micro_batch_size. That determines how many more requests the current microbatch can accept. Separately, PrefillAdder uses the 16,384-token budget and page-aligned chunking to decide how much of a long prompt is included in this extend pass.

So, for the current prefill chart, the expected behavior is up to six requests per PP microbatch, with each extend/chunk constrained by a 16,384-token prefill budget. A single long request can still consume the token budget and become the only request in that microbatch.
