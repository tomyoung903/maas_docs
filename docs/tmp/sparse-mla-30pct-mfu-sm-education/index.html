Sparse MLA at 30.84% MFU: inside one GPU SM

One 16,384-token GLM-5.2 sparse-attention launch becomes 16,384 thread blocks. Each block consumes an entire SM's register file and 94.5% of its shared-memory budget, leaving 16 resident warps to feed Tensor, load/store, reduction, and softmax work.

Answer first

The 30.84% number does not mean that 30.84% of the SMs are switched on. All 148 SMs receive work. It means the counted sparse-MLA useful FP8-equivalent operations arrive at 30.84% of a 4.5-PFLOP/s dense-FP8 comparison peak.

The standalone call reproduced the result. The production trace measured 30.84%. The direct synthetic call measured 31.58% at 3.083 ms. The difference is 0.74 percentage points, while preserving the exact API, dimensions, kernel specialization, grid, block, register, and shared-memory shape.

The complete tensor journey for this 16,384-token launch

This section follows the actual tensor contract from device memory, through 16,384 CUDA thread blocks and the four warp schedulers in each occupied SM, to the final BF16 tensor. It deliberately separates measured facts from mappings that the compiled kernel does not expose.

Measured API shape, profiler counter, or launch field Strong inference implied by matching row and grid cardinality Teaching quotient useful arithmetic, not a claimed lane map

1. The six allocations entering the call

The direct call allocated 2,121.0625 MiB. PyTorch recorded 2,121.0659 MiB at peak, a difference of only 3,584 bytes from the six explicit allocations below.

AllocationFootprintPer query rowConcrete meaning
Query, FP8[16,384, 1, 64, 576] 576 MiB 36 KiB For each token row: 64 head-specific query vectors. Each vector has a 512-element absorbed latent part and a 64-element RoPE part.
Compressed KV cache, FP8[256, 1, 64, 576] 9 MiB Shared by all heads 256 pages × one shared MLA KV stream × 64 tokens/page × 576 values. The first 512 values are compressed latent KV; the last 64 are RoPE key values.
Selected-token table, INT32[16,384, 1, 2,048] 128 MiB 8 KiB Physical KV-token indices for each query. Invalid causal-prefix slots are -1; valid count is min(t + 1, 2,048).
Sequence lengths, INT32[16,384] 64.0 KiB 4 bytes The valid selected-token count for each query row, capped at 2,048.
Workspace, UINT8[384 MiB] 384 MiB Kernel-wide scratch FlashInfer workspace supplied to the compiled TRT-LLM GEN implementation. Its internal subdivision is not exposed by the API or trace.
Output, BF16[16,384, 1, 64, 512] 1,024 MiB 64 KiB For each query row and head, 512 latent output values. There are 536,870,912 BF16 values in total.
Do not interpret the two visible 64s the same way. In the query tensor, 64 is the number of attention heads. In the KV tensor, 64 is the page size. MLA stores one compressed 576-value KV vector per physical token and shares it across all 64 query heads; it does not store 64 independent KV-head vectors.

2. Follow one query row t

Let Lt be min(t + 1, 2,048). For one head h, split the 576-value absorbed query into 512 latent values and 64 RoPE values. For every selected physical token j, the conceptual fused-attention calculation is:

scoret,h,j= 1 / sqrt(128 + 64) × (query-latentt,h · KV-latentj + query-RoPEt,h · key-RoPEj)
probabilityt,h,j= softmax over the Lt selected scores
outputt,h= sum over selected j of probabilityt,h,j × KV-latentj, producing 512 BF16 values
Why the dot product has 576 elements while the scale says 128 + 64: the API receives the model's original no-RoPE dimension qk_nope_head_dim=128 for scaling, but the query handed to this absorbed MLA kernel has already been transformed into the 512-dimensional compressed-KV latent space. Appending 64 RoPE values gives the observed 576-element QK operand.
Step 1 Own row t Read 36 KiB of FP8 query data: 64 vectors × 576 values.
Step 2 Resolve selections Read an 8 KiB INT32 row and its valid length, then translate selected physical tokens into paged-KV addresses.
Step 3 Gather KV tiles A full row names 2,048 × 576 FP8 values, a 1.125 MiB logical KV slice, so it must be consumed in tiles.
Step 4 QK and softmax Form 64 × Lt conceptual scores, scale them, and normalize each head's selected set.
Step 5 Probability × V Accumulate 512 latent output channels for each of the 64 heads using the same selected compressed KV vectors.
Step 6 Store row t Convert and write 32,768 BF16 values, exactly 64 KiB for this output row.

For a full 2,048-selection row, this is 131,072 selected head-pairs and 285.213 million counted FLOPs. Across the launch, the causal prefix produces 31,458,304 query-KV pairs and 2,013,331,456 selected head-pairs.

3. Fan 16,384 rows out across 148 SMs

Nsys and NCU both recorded grid (1, 1, 16,384) and block (512, 1, 1). The grid therefore contains one CTA for every flattened query row. The one-CTA-per-row ownership is a strong inference from equal cardinality and the API contract; the profiler cannot expose the private cubin's blockIdx.z expression.

Grid 16,384 CTAs 8,388,608 launched threads and 262,144 launched warps over the complete kernel.

Complete waves 110 110 × 148 = 16,280 blocks retire before the tail.
Final partial wave 104 CTAs The remaining 104 blocks occupy at most 104 SMs; scheduling order is not guaranteed.
Hierarchy boundaryWhat is exact hereWhat must not be overclaimed
Grid to SM 16,384 independent blocks are admitted onto 148 SMs, at one resident block per SM. CUDA does not promise that increasing block IDs run in increasing order.
Block to query row Grid cardinality exactly matches the 16,384 flattened query rows, strongly indicating one row owner per CTA. Nsys and NCU do not reveal the compiled blockIdx.z-to-row address expression.
Block to warps 512 threads divide exactly into 16 hardware warps of 32 lanes. The block cannot split across SMs or migrate while resident. A warp is an instruction-execution group, not necessarily a permanent attention-head owner.
Warp to heads An even-share quotient is 64 heads / 16 warps = 4 head-equivalents per warp. The private kernel may use warp specialization, cooperative head tiles, and producer/consumer roles. Four heads per warp is not a measured mapping.
Thread lanes to one head An even-share quotient is 512 threads / 64 heads = 8 thread-equivalents per head, then 512 output values / 8 = 64 values per thread-equivalent. This arithmetic is only a scale model. Tensor fragments, reductions, shuffles, and staged loads can assign entirely different lane roles.

4. What one admitted block does inside one SM

Admission immediately reserves all 65,536 32-bit registers and 220,672 bytes of shared memory for this block. The 512 threads form 16 warps; the SM's four schedulers see about four active warps each. A scheduler advances a warp one ready instruction at a time. It does not run one warp to completion before choosing another.

StageObjectConcrete movement or calculationWhere state livesCoordination
0CTA admission One 512-thread block receives an SM after the prior resident block retires. 16 warp contexts, 262,144 register bytes, and 220,672 shared-memory bytes are reserved. CUDA block scheduler; blocks are independent.
1Metadata Warps load the row length and selected INT32 physical-token indices, then derive paged-KV addresses. HBM-backed arrays arrive through L2 and L1/TEX; current indices and pointers reside in registers. Scoreboard dependencies prevent consumers from issuing before loads return.
2Q and KV tiles Warps fetch a 36 KiB query row and stream the up-to-1.125 MiB logical selected-KV slice in smaller tiles. Cache lines pass through L2/L1; operands and pipeline tiles occupy registers and shared memory. Kernel barriers or dependency protocols keep producers and consumers on the correct tile.
3QK dots Tensor-pipeline instructions consume FP8 query and KV fragments and accumulate 576-element dot products. Input fragments and partial sums are primarily register/shared-memory state; NCU measured 61.78% tensor-pipe active cycles. Warps issue only when operands and the required execution pipe are ready.
4Streaming softmax Per-head maxima and sums are reduced while selected tiles advance; scale is 1 / sqrt(192). Per-thread fragments use registers; cross-lane or cross-warp exchange can use shuffles and shared memory. Reduction dependencies and synchronization create non-Tensor instructions and wait states.
5Probability × V Normalized probabilities weight the 512-value compressed latent vectors and update each head's output accumulator. Tensor/ALU inputs and output accumulators are tiled through registers and shared memory. Compute overlaps with later tile movement only when an independent warp is eligible.
6BF16 output Final 512-value accumulators for 64 heads are converted and written as one 64 KiB output row. Register accumulators become global stores through L1/L2 to the 1 GiB output allocation. The block retires only after all its warps finish and stores are committed as required.
7Next wave Register and shared-memory allocations are released; another queued CTA can enter the SM. The departing block's register/shared state ceases to exist. L2 lines may remain reusable by later SMs. This repeats for 110.70 average waves per SM.
One thread 128 registers 512 bytes of allocated private register state for pointers, fragments, predicates, and accumulators.
One warp 16 KiB registers 32 lanes execute a common instruction under lane predicates; each lane still owns its private register values.
One block 256 KiB registers 16 warps collectively consume the entire 256 KiB SM register file plus 215.5 KiB shared memory.
Four schedulers 0.44 eligible each 3.99 active but only 0.37 issued warps per scheduler on an average cycle.
The exact 220,672-byte shared-memory layout is not public. It is defensible to say the fused kernel needs block-wide staging, reductions, barriers, and pipeline state. It is not defensible to label particular byte ranges as Q, K, V, probability, or accumulator buffers without source or SASS-level reverse engineering. The same limit applies to exact head-to-warp and output-dimension-to-lane maps.

5. Why logical work and physical traffic differ sharply

9 MiB KV pool

versus 126.5 MiB of measured GPU-wide L2

The complete synthetic physical KV pool fits in L2 more than fourteen times over.

The recent selection layout also makes adjacent full query rows share 2,047 of their 2,048 physical KV positions. This creates heavy cross-CTA address reuse even though the useful calculation still counts 2,013,331,456 selected head-pairs.

The memory path for an input tile is: HBM-backed tensor → GPU-wide L2 → per-SM L1/TEX → explicitly staged shared memory and thread-private registers → execution pipes. Output travels back through the cache hierarchy to its HBM-backed allocation. NCU measured 63.91% L1/TEX hit rate, 94.38% L2 hit rate, and only 7.45% of peak DRAM throughput.

That does not make the kernel free of memory latency. A miss or an irregular gather can still block a dependent warp while aggregate HBM bandwidth remains low. The scheduler evidence is direct: 63.4% of cycles had no eligible warp, and long-scoreboard waits contributed 7.47 of the measured 10.91 warp-cycles per issued instruction.

The complete causal chain behind 30.84% MFU. One resource-heavy CTA fills each SM with only 16 resident warps. Those warps cooperatively gather selected KV tiles, execute QK, normalize scores, accumulate probability-times-V, and store BF16 output. When their load and reduction dependencies align, the four schedulers have no independent ready warp to hide the wait. Tensor work is substantial, but tile movement, softmax, synchronization, conversion, and empty issue opportunities all consume elapsed time outside the 4.381-TFLOP useful numerator.

What the 16 warps do each cycle

A resident warp is merely available to the scheduler. It is eligible only when its next instruction is decoded, its input dependencies are ready, and the required execution pipe is available. NCU measured 3.99 active warps per scheduler but only 0.44 eligible on an average cycle.

Total: 10.91 warp-cycles per issued instruction. The dominant 68.5% is waiting for an L1TEX-tracked dependency, usually a value produced by a local/global/texture load.

Low occupancy is not the same as low MFU. Sixteen warps can sometimes drive Tensor Cores effectively. Here, however, the four active warps per scheduler often wait on the same class of dependency, so there is no independent ready warp to issue on 63.4% of scheduler cycles.

Exactly how 30.84% arises

For the first 2,048 causal queries, the number of valid selected keys grows from 1 to 2,048. The remaining 14,336 queries each use 2,048 keys. Every selected query-key pair is evaluated for 64 heads.

selected pairs= 2,048 × 2,049 / 2 + (16,384 - 2,048) × 2,048 = 31,458,304
FLOPs / pair / head= 2 × (512 + 64) + 2 × 512 = 2,176
useful work / layer= 31,458,304 × 64 × 2,176 = 4,381,009,248,256 FLOPs
production MFU= 43.810092 TFLOP / (0.031566236 s × 4,500 TFLOP/s) = 30.8417%
synthetic MFU= 4.381009 TFLOP / (0.003083094 s × 4,500 TFLOP/s) = 31.5773%
What the numerator intentionally excludes: sparse-index generation, page-table reads, softmax exponentials, reductions, synchronization, and BF16 output conversion. They are necessary execution, but the established MFU convention counts the FP8-equivalent Q·K and probability·V operations only.

These register and shared-memory values are arithmetic upper bounds from the measured per-SM budgets. Register allocation granularity, barriers, and other launch constraints can lower actual occupancy.

More occupancy is an opportunity, not a speedup formula. Reducing registers or shared memory can admit more blocks, but those blocks help only if they provide eligible warps while another warp is waiting. Extra blocks may also reduce cache locality or force spills. Interpretation limits

  • The cluster calls the fleet B300, while the device reports NVIDIA L20D, compute capability 10.3, and 148 SMs.
  • The 4,500-TFLOP/s dense FP8 denominator is a comparison convention derived from NVIDIA's DGX B300 system specification, not a public L20D product specification.
  • Synthetic selected-token indices preserve cardinality and launch geometry, but they cannot reproduce the exact learned DSA index distribution or the complete SGLang layer context.
  • NCU replay changes wall time. NCU counters describe the replayed kernel; unprofiled CUDA events and Nsys provide timing.
  • The TensorRT-LLM GEN kernel is a compiled specialization. The trace proves launch and resource geometry, but it does not expose the exact head-to-warp, dimension-to-lane, register, or shared-memory layout. Even-share quotients are teaching aids, not reverse-engineered implementation claims.
