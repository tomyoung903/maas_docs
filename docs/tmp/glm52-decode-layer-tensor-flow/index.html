D1/D2 and C1/C2 are the two observed kernels in each logical DeepEP phase. The trace does not expose enough semantic metadata to assign a more specific role to each member.

What actually overlaps? The shared-expert branch runs on a separate stream and overlaps DeepEP dispatch. On one GPU, the routed chain itself remains ordered: dispatch → gate/up → activation → down → combine.

The physical DeepEP tensor is padded and expert-major. For this exact EP=8 capture, each rank owned 32 routed experts and reserved 512 slots per local expert. Dispatch produced an FP8 activation buffer in ℝ32 × 512 × 6,144, FP32 per-group scales in ℝ32 × 512 × 48, and 32 valid-row counts. How to read apparent overlap correctly

The chart has two different notions of concurrency: genuine same-rank stream overlap and aggregate overlap caused by eight ranks reaching the same phase at different times.

One GPU: routed path is ordered

Dispatch finishes before that GPU’s routed gate/up GEMM. Down finishes before that GPU’s combine. There were zero violations across 1,200 audited MoE instances.

One GPU: shared work overlaps

The shared expert uses an alternate stream and overlaps DeepEP dispatch by a median 14.657 µs. This is intentional compute/communication concurrency.

Eight GPUs: phases can overlap globally

A fast rank may reach expert compute while a slow rank is still dispatching. That is cross-rank skew, not a broken dependency on either rank.

Combine is also a synchronization boundary. Expert-load imbalance expanded the median eight-rank skew to 32.4 µs at routed-down completion. DeepEP combine reduced final skew to 5.4 µs, so long combine residency can include waiting rather than proportional useful transfer.

Evidence and limits

This note is a readable companion to the complete kernel timeline, not a replacement for it.

What is directly supported by the trace?
  • The listed timestamps, stream placement, kernel ordering, and overlap intervals.
  • Two DeepEP dispatch kernels, two routed expert GEMMs, and two DeepEP combine kernels per sparse layer.
  • Seventy-five sparse layers, eight GPUs, and two graph waves: 1,200 MoE layer instances.
  • The logical and physical dimensions established from model configuration, capture settings, runtime code, and GEMM kernel signatures.
  • The second graph wave is cleanup work, not a second GPU-computed token.
What should not be inferred?
  • The example hidden values, expert IDs, router weights, and expert outputs were not captured.
  • The trace does not reveal the private semantic division between the first and second DeepEP kernel in each pair.
  • The conceptual eight-row routed stack is not the full shape of DeepEP’s expert-major, capacity-padded communication buffers.
  • Summed GPU time across eight ranks is not the same quantity as elapsed wall time.
