Overview

Audit verdict. The raw profile is usable, but the original page mixed three different semantics. NCCL SendRecv duration is aggregate kernel residency rather than useful model compute; nanosecond log timestamps failed to parse, leaving the scheduler charts empty; and PP=8 duplicated logical token counts across eight stage logs. The charts and labels below now separate those quantities.

Run Shape

Standalone GLM-5.2 FP8 on gpu-worker-204, with PP=8, TP=1, DP=1, and max_tokens=1. This is prefill-shaped, not a disaggregated PD KV-transfer run.

GPU Evidence

78,946 visible kernel rows span 117.75 s. Their durations sum to 119.01 aggregate GPU-s across eight devices; that sum is not wall time.

Scheduler Evidence

The fixed parser finds 72 PP-stage prefill rows, 9 PP0 logical prefill entries, 17 receives, 1 finish, and 0 decode rows.

Result Boundary

The direct one-token request finished, but Locust recorded no completed request rows under profiler overhead. This report supports kernel and scheduler analysis; it does not provide a valid throughput or latency result.

Published artifacts: timeline_data.json and kernel_summary.csv.

Expanded Kernel Residency Density

Corrected interpretation. The blue area is mostly long-lived ncclDevKernel_SendRecv residency. 30 launches last at least one second and contribute 101.06 aggregate GPU-s. Overlap across devices means a one-second bin can contain several GPU-seconds. This is pipeline communication and synchronization residency, not 107.44 seconds of model compute.

Each kernel duration is split across every one-second bin it overlaps. The category remains nccl_comm in the downloadable data so the raw grouping stays auditable.

Kernel Launch Count Density

This chart counts the start bin of each visible kernel row. A long SendRecv kernel can add residency to later time bins without adding another launch there, so launch count and residency density are intentionally different views.

Scheduler Logs

Parser correction. The log uses nanosecond RFC3339 prefixes such as 2026-07-09T18:48:28.225530692+08:00. The original parser passed all nine fractional digits to a datetime implementation that accepts six, so every timestamp became null and both charts rendered as empty axes. The revised parser preserves ordering while truncating to microseconds.

Known limitations

  • This was a standalone prefill-shaped run on gpu-worker-204, not a full disaggregated PD bootstrap/KV-transfer test.
  • The Locust client produced zero completed rows in its 30 second window under Nsight overhead, so throughput and latency are unavailable.
  • A direct /generate check with max_new_tokens=1 completed and is recorded in the SGLang Finish log.
  • PP=8 emits one Prefill log row per pipeline stage; logical token totals and token charts use PP0 only.
  • The scheduler and CUPTI charts use separate time bases and must not be overlaid as if their t=0 timestamps match.
  • CUPTI kernel rows have no graphId/graphNodeId values, decode graphs were disabled, and the scheduler has zero Decode batch rows.
  • Aggregate kernel and memory-operation durations are summed across devices and are not wall-clock time.
  • NCCL SendRecv residency includes communication and synchronization waiting; it is not a model-compute measurement.

Kernel t=0 is the first CUPTI kernel event. Scheduler t=0 is the first timestamped Receive row; the two timelines have separate clocks and are not overlaid.
