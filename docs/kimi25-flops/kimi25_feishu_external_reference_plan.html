Kimi K2.5 Feishu External Reference Plan

Goal

The current Feishu doc is mathematically useful, but too abstract for readers who do not already understand MLA, absorbed attention, SwiGLU, MoE, and attention-pair accounting.

We should turn it into a learning path:

  1. Keep the Feishu doc as the narrative spine.
  1. Host the detailed HTML and notebook artifacts at stable HTTPS URLs.
  1. Insert those URLs into Feishu as preview/card blocks near the relevant sections.
  1. Let readers open the visual explanations or executable notebooks only when they need more depth.

Why This Is Better Than Only Attaching Files In Feishu

Feishu links can display as link/title/card/preview views. If the URL points to a stable HTTPS page, Feishu can show a usable preview or card in the doc.

That gives us:

  • one Feishu doc as the main narrative;
  • inline previews/cards for visual material;
  • stable URLs for teammates to share;
  • source files still preserved in the repo;
  • no need to paste huge diagrams or notebooks directly into Feishu.

Relation To The Existing ntfy Pattern In ~/armory

The ntfy setup in ~/armory is useful as an operational pattern:

  • config lives in ~/armory/.shared/noti_ntfy.env;
  • scripts read the service endpoint from env vars;
  • a small service is managed separately from the repo content.

For this Kimi docs case, we should reuse the same philosophy, not the same app:

  • define one env/config value like MAAS_DOCS_BASE_URL;
  • publish static files to a small internal docs service;
  • generate Feishu links from that base URL;
  • keep source artifacts in z_local.

Hosting Options

Option A: Internal static docs service

Run Nginx or Caddy on a small internal VM.

Example shape:

/srv/maas-docs/kimi25-flops/

Pros:

  • simplest runtime;
  • static HTML and notebook HTML load fast;
  • easy to add basic auth or company network restrictions;
  • good enough for internal learning material.

Cons:

  • someone owns the VM and HTTPS config.

This is my recommended first implementation.

Option B: Kubernetes static-site Deployment

Build a tiny Nginx image or ConfigMap-backed static site and expose it through an internal Ingress.

Pros:

  • fits our existing Kubernetes operations;
  • easy to version by image tag;
  • can use existing ingress/auth patterns.

Cons:

  • more operational overhead than a VM static folder;
  • ConfigMap size may be annoying if artifacts grow.

Use this if we want the site managed like production infra.

Option C: GitLab Pages or object storage

If internal GitLab Pages or object storage with HTTPS exists, publish the artifact directory there.

Pros:

  • very low maintenance;
  • versioning is easy;
  • source and published artifacts can be tied to commits.

Cons:

  • depends on whether the feature is available internally;
  • access control may be less convenient.

Use this if the platform already exists.

Option D: Feishu file attachments only

Upload HTML/notebook files as Feishu attachments.

Pros:

  • fastest manual fallback.

Cons:

  • weak inline experience;
  • hard to update;
  • not good for interactive HTML previews.

Use only as a temporary fallback.

Implementation Steps

  1. Create a publish directory:
mkdir -p z_local/benchserving/mfu_metrics/publish/kimi25-flops
  1. Copy the existing HTML files into it.
  1. Convert notebooks to rendered HTML:

  z_local/benchserving/mfu_metrics/kimi25_linear_flops_constant.ipynb \
  --output kimi25_linear_flops_constant.notebook.html \
  --output-dir z_local/benchserving/mfu_metrics/publish/kimi25-flops

Feishu API Notes

The manual path is simple: paste URLs into Feishu and choose Link View, Title View, Card View, or Preview View.

The API path may be possible with Docx blocks:

  • link previews are supported by Feishu Docs;
  • the Docx block model includes an iframe / embedded block type;
  • actual behavior still needs a small probe against our hosted URL, because generic external websites may be restricted by Feishu or by the target page's headers.

For reliability, start with hosted URLs plus manual Preview/Card selection. Then automate iframe/link-preview insertion only after a probe works.
