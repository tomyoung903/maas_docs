Kimi K2.5 架构到模型 FLOPs 拆解

这篇文档接在《模型 FLOPs 拆解教程》后面。上一篇讲的是通用框架：

model_FLOPs = linear_FLOPs + attention_FLOPs

这一篇只讲 Kimi K2.5：怎么从 Kimi 的模型结构出发，得到两个常数：

  • linear_FLOPs_per_forward_token = 61,023,322,112
  • attention_FLOPs_per_attention_pair = 8,495,104

这两个常数一旦有了，任何 serving 请求都可以用下面的方式计算：

model_FLOPs = forward_token_count * linear_FLOPs_per_forward_token + attention_pair_count * attention_FLOPs_per_attention_pair

本文先不考虑 TP、通信、内存带宽、量化吞吐差异、softmax、RMSNorm、采样和 lm_head。这里算的是一个单一逻辑模型上的主要矩阵乘 FLOPs，乘加按 2 FLOPs 计。

1. Kimi K2.5 的核心结构

Kimi K2.5 的 text tower 可以先按一个 transformer block 来理解。

每个 token 进入一层时，主要会经过两大块：

  1. MLA attention 相关计算。
  1. MLP / MoE feed-forward 相关计算。

Kimi K2.5 一共有：

  • L = 61 层
  • hidden size H = 7168
  • attention heads = 64
  • 第 0 层是 dense MLP
  • 第 1 到 60 层是 MoE
  • MoE 有 384 个 routed experts
  • 每个 token 选择 topk = 8 个 routed experts
  • 还有 shared_experts = 1 个共享 expert
  • 所以每个 token 实际运行 active_experts = 9 个 experts

最重要的直觉是：Kimi 虽然有 384 个 routed experts，但每个 token 不会跑完 384 个 expert MLP。它只跑 8 个 routed experts 加 1 个 shared expert。这就是 sparse-MoE 的核心节省。

2. 为什么先拆成 linear 和 attention scan

Kimi 的 FLOPs 也先拆成两类：

model_FLOPs = linear_FLOPs + attention_FLOPs

linear_FLOPs 是每个真正 forward 的 token 都要付一次的模型层计算。

attention_FLOPs 是 query token 回看上下文时产生的 attention scan 计算。

这两个东西必须分开，因为它们增长方式不同：

  • linear_FLOPs 随 forward_token_count 近似线性增长。
  • attention_FLOPs 随 attention_pair_count 增长。
  • cached prompt tokens 通常不再付完整 linear FLOPs。
  • cached prompt tokens 仍然在上下文里，所以可能继续贡献 attention pairs。

所以 Kimi 的 serving FLOPs 可以写成：

Kimi_model_FLOPs = forward_token_count * 61,023,322,112 + attention_pair_count * 8,495,104

下面解释这两个常数怎么从架构算出来。

3. Kimi 的 MLA attention 投影部分

Kimi 使用 MLA。MLA 的一个核心目的，是把 KV cache 压到 latent 维度里，而不是为每个 head 存完整 K/V。

本文使用的 Kimi MLA 维度是：

  • q_lora = 1536
  • kv_lora = 512
  • qk_nope = 128
  • qk_rope = 64
  • qk_head = qk_nope + qk_rope = 192
  • v_head = 128
  • heads = 64

对一个 forward token，在一层 attention 里，投影部分包括：

  • qkv_a
  • q_b
  • W_KC
  • W_VC
  • o_proj

这些都是每个 forward token 都要做的矩阵乘，所以它们属于 linear_FLOPs，而不是 attention_FLOPs。

3.1 qkv_a

qkv_a 从 hidden state 一次性投影出 query latent、KV latent 和 key-side RoPE 分支：

qkv_a = 2 * H * (q_lora + kv_lora + qk_rope)

代入 Kimi 数值：

qkv_a = 2 * 7168 * (1536 + 512 + 64) = 30,277,632

这里的三个输出分别是：

  • q_lora：query 的低秩 latent
  • kv_lora：KV cache 的 latent content
  • qk_rope：key-side positional branch

3.2 q_b

q_b 把 query latent 展开成所有 head 的 query：

q_b = 2 * q_lora * (heads * qk_head)

代入 Kimi 数值：

q_b = 2 * 1536 * (64 * 192) = 37,748,736

这里 qk_head = 192，是 qk_nope = 128 和 qk_rope = 64 拼起来。

3.3 W_KC

W_KC 是 absorbed MLA 里的 content-key 相关投影。

直觉上，普通写法会把 kv_lora = 512 展开成每个 head 的 content key，维度是 qk_nope = 128。SGLang 的 absorbed MLA 不需要把这些 full key 全部物化到 cache 里，而是把这部分矩阵吸收到 query 侧。

FLOPs 记账上：

W_KC = 2 * heads * qk_nope * kv_lora

代入 Kimi 数值：

W_KC = 2 * 64 * 128 * 512 = 8,388,608

注意：这个 FLOP 数和“显式展开 content key”的算术量等价，但 absorbed 的收益主要是 runtime memory traffic 和 KV cache 形态更好，不是这一个投影本身的 FLOP 数变少。

3.4 W_VC

W_VC 是 absorbed MLA 里的 value 相关投影。

attention scan 先在 kv_lora = 512 的 latent 空间里累加 value，再用 W_VC 映射到每个 head 的 v_head = 128。

FLOPs 记账上：

W_VC = 2 * heads * kv_lora * v_head

代入 Kimi 数值：

W_VC = 2 * 64 * 512 * 128 = 8,388,608

3.5 o_proj

o_proj 把所有 heads 的 attention 输出投影回 hidden size：

o_proj = 2 * (heads * v_head) * H

代入 Kimi 数值：

o_proj = 2 * (64 * 128) * 7168 = 117,440,512

3.6 attention projection per layer

所以每层 attention 投影部分是：

attention_projection_per_layer = qkv_a + q_b + W_KC + W_VC + o_proj

代入数值：

attention_projection_per_layer = 30,277,632 + 37,748,736 + 8,388,608 + 8,388,608 + 117,440,512 = 202,244,096

这个 202,244,096 是每层、每个 forward token 的 attention projection FLOPs。它还不是 attention scan FLOPs。

4. Kimi 的 MLP / MoE / router 部分

Kimi 的 feed-forward 结构分两种：

  • 第 0 层是 dense SwiGLU MLP。
  • 第 1 到 60 层是 sparse MoE。

4.1 Dense MLP

Dense SwiGLU MLP 有 gate、up、down 三个主要矩阵乘：

dense_mlp_layer = 6 * H * dense_intermediate

Kimi 的 dense_intermediate = 18432：

dense_mlp_layer = 6 * 7168 * 18432 = 792,723,456

这一项只出现 1 层。

4.2 Router

每个 MoE layer 先用 router 给 384 个 routed experts 打分：

router_per_moe_layer = 2 * H * routed_experts

代入 Kimi 数值：

router_per_moe_layer = 2 * 7168 * 384 = 5,505,024

router 会看所有 384 个 routed experts，但这只是一个小矩阵乘，不等于跑 384 个 expert MLP。

4.3 Active MoE experts

一个 expert 本身也是 SwiGLU MLP：

one_expert = 6 * H * moe_intermediate

Kimi 的 moe_intermediate = 2048：

one_expert = 6 * 7168 * 2048 = 88,080,384

每个 token 实际跑：

active_experts = topk + shared_experts = 8 + 1 = 9

所以每个 MoE layer 的 active expert FLOPs 是：

active_moe_layer = active_experts * one_expert

代入数值：

active_moe_layer = 9 * 88,080,384 = 792,723,456

注意这里没有乘 384。384 是候选 routed experts 数量，不是每个 token 实际运行的 expert 数量。

5. Kimi 的 linear_FLOPs_per_forward_token

现在把每个 forward token 的所有 linear 部分加起来。

Kimi 有 61 层 attention projection：

61 * attention_projection_per_layer = 61 * 202,244,096 = 12,336,889,856

Kimi 有 1 层 dense MLP：

1 * dense_mlp_layer = 792,723,456

Kimi 有 60 层 active MoE：

60 * active_moe_layer = 60 * 792,723,456 = 47,563,407,360

Kimi 有 60 层 MoE router：

60 * router_per_moe_layer = 60 * 5,505,024 = 330,301,440

所以：

linear_FLOPs_per_forward_token = 12,336,889,856 + 792,723,456 + 47,563,407,360 + 330,301,440

最终得到：

linear_FLOPs_per_forward_token = 61,023,322,112

这个数的意思是：一个 token 真正跑一次 Kimi text tower forward 时，大约要付 61.0B 的主要线性层 FLOPs。

6. Kimi 的 attention scan FLOPs

上面算的是每个 forward token 都要做的 projection / MLP / MoE 工作。

但 attention 还有另一部分：query token 要回看上下文。

我们把“一个 query token 看一个 context token”叫做一个 attention pair。

Kimi absorbed MLA 的 scan 对每个 attention pair、每层、所有 heads 做三类主要计算：

  1. content QK：latent content query 和 cached latent content 做 dot。
  1. RoPE QK：query RoPE branch 和 key RoPE branch 做 dot。
  1. AV latent accumulation：用 attention weight 累加 latent value。

对于每层：

content_qk_per_layer = heads * 2 * kv_lora
rope_qk_per_layer = heads * 2 * qk_rope
av_latent_per_layer = heads * 2 * kv_lora

代入 Kimi 数值：

content_qk_per_layer = 64 * 2 * 512 = 65,536
rope_qk_per_layer = 64 * 2 * 64 = 8,192
av_latent_per_layer = 64 * 2 * 512 = 65,536

所以：

attention_scan_FLOPs_per_pair_per_layer = 65,536 + 8,192 + 65,536 = 139,264

Kimi 有 61 层：

attention_FLOPs_per_attention_pair = 61 * 139,264 = 8,495,104

这个数的意思是：一个 query token 多看一个 context token，就要多付约 8.5M 的 Kimi attention scan FLOPs。

7. 为什么 qk_nope 和 v_head 不在 scan 常数里

这点容易混淆。

qk_nope = 128 不直接出现在 attention scan constant 里，是因为 absorbed MLA 已经把 no-position query 通过 W_KC 映射到了 kv_lora = 512 的 latent content 空间。W_KC 这个投影已经算在 linear_FLOPs 里。

v_head = 128 也不直接出现在 attention scan constant 里，是因为 scan 先累加 kv_lora = 512 的 latent value。后续通过 W_VC 映射到 v_head = 128。W_VC 也已经算在 linear_FLOPs 里。

所以 scan constant 里看到的是：

2 * heads * (2 * kv_lora + qk_rope)

也就是：

2 * 64 * (2 * 512 + 64) = 139,264 per layer

8. 回到 serving request

有了这两个 Kimi 常数后，serving 请求的模型 FLOPs 就可以写成：

Kimi_model_FLOPs = forward_token_count * 61,023,322,112 + attention_pair_count * 8,495,104

其中：

  • forward_token_count = uncached_prefill_tokens + decode_forward_tokens
  • attention_pair_count = prefill_attention_pairs + decode_attention_pairs

cached prompt tokens 的处理方式是：

  • cached prompt tokens 通常不再重复跑完整模型，所以不重复计入 forward_token_count。
  • cached prompt tokens 仍然在上下文里，所以可能继续计入 attention_pair_count。

这就是为什么实际 serving 效率分析里，不能只看 input tokens 和 output tokens。要进一步拆成 uncached prefill、cached prefix、decode forward 和 attention pairs。

9. 关键公式汇总

  • model_FLOPs = linear_FLOPs + attention_FLOPs

模型 FLOPs 先拆成每个 forward token 固定要付的 linear 部分，以及随上下文长度增长的 attention scan 部分。

  • linear_FLOPs = forward_token_count * linear_FLOPs_per_forward_token

Kimi 的 linear 常数是 61,023,322,112。

  • attention_FLOPs = attention_pair_count * attention_FLOPs_per_attention_pair

Kimi 的 attention pair 常数是 8,495,104。

  • linear_FLOPs_per_forward_token = 61 * attention_projection_per_layer + 1 * dense_mlp_layer + 60 * active_moe_layer + 60 * router_per_moe_layer

这条公式把 Kimi 的 attention projection、dense MLP、MoE experts 和 router 合在一起。

  • attention_projection_per_layer = qkv_a + q_b + W_KC + W_VC + o_proj

这是 Kimi MLA 中每层、每个 forward token 的 projection 部分。

  • attention_FLOPs_per_attention_pair = 61 * 2 * heads * (2 * kv_lora + qk_rope)

这是 Kimi absorbed MLA 中每个 attention pair 的 scan 部分。
