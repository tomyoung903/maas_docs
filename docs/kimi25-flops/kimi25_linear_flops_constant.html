Kimi K2.5 Linear FLOPs Constant

This page justifies the no-parallelism baseline constant: 61,023,322,112 FLOPs per forward token.

Scope contract

Single giant GPU, no parallelism, multiply-add counted as 2 FLOPs, Kimi K2.5 active transformer-block math. Router is included. Attention scan, softmax, RMSNorm, RoPE elementwise work, sampling, communication, and lm_head are excluded.

Architecture Diagram: Where Linear FLOPs Live

For this constant, imagine one forward token flowing through the Kimi K2.5 text transformer. The diagram highlights matmul work charged once per uncached prompt token or output token. It deliberately excludes the context-length attention scan, which is counted by the separate pair-based constant.

Forward token hidden[7168] One uncached prompt token or one decode/output token enters every transformer block.
->
61 transformer blocks Each block has MLA attention projections plus a feed-forward path.
MLA projection side qkv_a, q_b, W_KC, W_VC, o_proj Charged as 202,244,096 FLOPs/layer.
Feed-forward side dense MLP or router + active MoE Layer 0 uses dense MLP. Layers 1-60 use top-8 routed experts plus one shared expert.
->
Not counted here QK/AV scan over context Those FLOPs scale with attention pairs and use the other constant: 8,495,104 FLOPs/pair.

Mental model: linear FLOPs are mostly per-token matrix multiplications inside each block; attention-scan FLOPs are per query-context pair and grow with prompt/cache length.

2. Attention Projection FLOPs Per Layer

Kimi's served MLA path uses qkv_a, all-head q_b, absorbed W_KC and W_VC, and o_proj. In this baseline all heads are on one giant GPU.

Component Formula FLOPs Ownership
qkv_a 2 * 7168 * (1536 + 512 + 64) 30,277,632 One copy on the single giant GPU.
q_b 2 * 1536 * (64 * 192) 37,748,736 All 64 heads are included.
absorbed W_KC 2 * 64 * 128 * 512 8,388,608 Across all heads, maps content query into latent KV space.
absorbed W_VC 2 * 64 * 512 * 128 8,388,608 Across all heads, maps latent attention output back to value-head space.
o_proj 2 * (64 * 128) * 7168 117,440,512 All attention-output dimensions are included.
Total sum above 202,244,096 Per layer, per token.

Why no separate kv_b row? The code defines kv_b_proj over the heads. In the absorbed MLA path, that same kv_b weight is algebraically split into W_KC for content-score absorption and W_VC for value-output absorption. The combined FLOPs are identical: 2 * 512 * (64 * (128 + 128)) = 16,777,216, which equals 8,388,608 + 8,388,608. We show the split form because it matches the runtime mental model: one part moves the query into latent KV space before the scan, and the other maps the latent attention output back afterward.

3. MLP, MoE, and Router FLOPs

A SwiGLU MLP has gate, up, and down projections, so its dense matmul cost is 6 * hidden * intermediate. With no parallelism, the full dense and expert intermediate dimensions are on the single giant GPU.

Component Formula FLOPs Reason
Dense MLP layer 6 * 7168 * 18432 792,723,456 Only layer 0 is dense because first_k_dense_replace=1.
Active MoE layer 6 * 7168 * 2048 * 9 792,723,456 Each token runs top-8 routed experts plus 1 shared expert.
Router per MoE layer 2 * 7168 * 384 5,505,024 Router scores all 384 routed experts.

The dense MLP and active MoE layer happen to have the same FLOPs because 18,432 = 2,048 * 9. This equality is not a general law; it is specific to Kimi K2.5's dimensions and top-8 + 1 shared expert choice.

4. Final Sum

linear_flops_per_token =
  61 * attention_projection_per_layer
  + 1 * dense_mlp_layer
  + 60 * active_moe_layer
  + 60 * router_per_moe_layer

= 61 * 202,244,096
  + 1 * 792,723,456
  + 60 * 792,723,456
  + 60 * 5,505,024

= 12,336,889,856
  + 792,723,456
  + 47,563,407,360
  + 330,301,440

= 61,023,322,112 FLOPs/token

5. What This Constant Does Not Include

Excluded Why excluded here
Attention scanSeparated into the pair-based constant 8,495,104 FLOPs/context-pair.
RMSNorm, RoPE, softmax, top-k, samplingSmaller non-matmul operations; not included in this theoretical model-FLOP constant.
CommunicationAll-reduce/all-to-all affects runtime, not this model arithmetic count.
lm_headExcluded to keep the constant focused on transformer-block active FLOPs.
Quantized bit-opsThe count is BF16-equivalent logical multiply-add FLOPs, not W4A16 kernel throughput or byte traffic.
