MLA Attention, From Zero To Kimi TP=8

This page deliberately slows down the explanation. First: normal attention. Second: why MLA exists. Third: Kimi K2.5 MLA as one logical model. Last: how TP=8 maps that logical model onto GPUs.

Scope

This is only about MLA attention and TP=8. Sections 1-6 use global 64-head shapes. Section 7 converts those shapes to per-rank execution. MoE is intentionally left out except where it touches the hidden stream before/after attention.

1. Background: Normal Attention

In normal attention, every token creates a query Q, a key K, and a value V. The query asks: "which previous tokens should I read?" Keys are addresses. Values are the content to read.

For one head, one query token i: q_i = hidden_i x W_Q k_j = hidden_j x W_K for every cached token j v_j = hidden_j x W_V for every cached token j score_ij = dot(q_i, k_j) / sqrt(d_head) score_ij += causal_mask(i, j) p_ij = softmax(score_i over all j) out_i = sum_j p_ij * v_j

Softmax is simple: for a fixed query token and fixed head, it turns all context scores into probabilities that sum to 1. High score means "read more from that old token".

2. Why MLA Exists

The KV-cache problem

Decode repeatedly scans all cached keys/values. If each token stores full per-head K and V, the cache is huge and memory bandwidth becomes the bottleneck.

MLA idea

Store a compressed latent vector instead of full K/V. Later, use learned projections to get the same useful attention computation from this smaller cache.

Kimi shape

Kimi stores normalized latent c/k_nope[512] plus RoPE-applied k_rope[64] per token, rather than full per-head K/V for every head.

Normal cache, conceptual: for each old token j: K_j: [num_heads, d_key] V_j: [num_heads, d_value] MLA cache, Kimi-style: for each old token j: c_j / k_nope_j: [512] # normalized compressed content latent k_rope_j: [64] # RoPE-applied position key MLA saves cache/memory traffic because c_j is shared across heads and reused through learned head-specific transforms.

3. Kimi MLA: What Is Stored And What Is Rebuilt?

Ignore tensor parallelism for now. Think of one logical Kimi attention layer. In SGLang, qkv_a is a fused projection: the Moonshot reference model has separate q_a_proj and kv_a_proj_with_mqa, while SGLang fuses them for serving. The raw KV content is normalized before it is used as the cached latent c_j, which SGLang often calls k_nope.

Kimi K2.5 dimensions: hidden = 7168 q_lora_rank = 1536 kv_lora_rank = 512 qk_nope_head_dim = 128 qk_rope_head_dim = 64 v_head_dim = 128 num_heads = 64 SGLang fused qkv_a: hidden[7168] -> q_lora_raw[1536] + kv_content_raw[512] + k_pe_raw[64] total output = 2112 Content path, used in the next section: q_lora = RMSNorm(q_lora_raw) # [1536] c_j / k_nope_j = RMSNorm(kv_content_raw) # [512] q_b, after q_lora normalization: q_lora[1536] -> 64 heads x (q_nope[128] + q_pe_raw[64]) global query output before RoPE = 64 x 192 = 12288 Position path, parked until section 5: q_pe_raw_h -> RoPE(position_i) -> q_rope_h # [64], query-side position k_pe_raw_j -> RoPE(position_j) -> k_rope_j # [64], cached key-side position qk_nope means "query/key dimensions with no positional encoding". qk_rope_head_dim is the shared dimension name: both q_rope and k_rope are length 64. Important: q_rope comes from q_lora -> q_b -> q_pe_raw -> RoPE. It is query-side and per head. k_rope comes from qkv_a -> k_pe_raw -> RoPE. It is key-side, cached, and shared across heads. the 128-d key-side no-RoPE vector can be rebuilt from c_j through kv_b_proj/w_kc, but absorbed MLA does not cache or materialize it.

Naming trap: qk_rope_head_dim=64 is a dimension setting, not a tensor source. The actual query vector is q_rope; the actual cached key vector is k_rope. Also do not confuse SGLang's cached k_nope[512] latent with qk_nope_head_dim=128. They meet only through the absorbed score computation.

4. MLA Without Position: Content-Only Attention

First remove positional embeddings from the mental model. Then MLA is just this: each old token stores a small content latent c_j/k_nope_j[512]. For the current token, each head builds a content query that can dot that latent. This gives one content score per old token.

For one logical attention head h, ignoring position: q_lora_raw = qkv_a(hidden_i).q_lora_raw # [1536] q_lora = RMSNorm(q_lora_raw) # [1536] q_all_raw = q_b(q_lora) # [64 heads, 192] q_h_raw = q_all_raw[h] # [192] q_nope_h, _ = split(q_h_raw, [128, 64]) # ignore the 64-d position slice for now For old token j in context: c_j / k_nope_j = cached normalized latent [512] Absorbed content-only MLA: q_content_h = q_nope_h x W_KC_h # [512], W_KC_h comes from kv_b_proj/w_kc score_content_j = dot(q_content_h, c_j/k_nope_j) score_j = score_content_j * scale Then: prob_j = softmax(score_j over all context positions j) latent_out_h = sum_j prob_j * c_j/k_nope_j # [512]

Mental model before position: the head asks, "which old token has content that matches my current content query?" It does not yet know whether an old token is near, far, earlier, or later. That positional information is added as a second score term in the next section.

5. Add Position: RoPE Contribution

Now add positional embeddings back. Kimi keeps the content score from section 4, then adds a separate positional score. The query-side position vector q_rope_h comes from q_lora -> q_b -> RoPE. The key-side position vector k_rope_j comes from qkv_a -> RoPE and is cached with the old token. Both are length 64.

Continue from the content-only score: score_content_j = dot(q_content_h, c_j/k_nope_j) Now add position: _, q_pe_raw_h = split(q_h_raw, [128, 64]) q_rope_h = RoPE(q_pe_raw_h, position_i) # current-token position query For old token j in context: k_rope_j = cached RoPE(k_pe_raw_j, position_j) Position term: score_pos_j = dot(q_rope_h, k_rope_j) Full MLA score: score_j = (score_content_j + score_pos_j) * scale score_j += causal_mask(i, j) Then: prob_j = softmax(score_j over all context positions j)

Why separated this way: content says "what old token looks relevant?" Position says "where is that old token relative to this query?" MLA keeps these as two additive score terms, which is why the content compression can be understood before RoPE.

6. After Softmax: How Values Are Produced

In normal attention, softmax probabilities multiply full V_j. In absorbed MLA, probabilities multiply the cached latent c_j / k_nope_j, producing a latent output. Then the value half of kv_b_proj, called w_vc in SGLang, converts that latent output into the real per-head value vector.

After softmax: latent_out_h = sum_j prob_j * c_j/k_nope_j # [512] value_out_h = latent_out_h x W_VC_h # [128] For the logical 64-head model: 64 heads x 128 = 8192 attention dims Then: attn = concat(value_out_h for 64 heads) # [8192] final_attn_output = attn x W_O # [7168]

7. MLA With TP=8 In Kimi K2.5

The easiest way to think about TP is: each GPU owns 8 heads. Softmax is local per head, so each GPU computes complete softmaxes for its 8 heads. There is no cross-GPU softmax. In the non-DP-attention setup, the 512+64 KV cache state is duplicated across TP ranks; the head-specific query/value work is sharded by heads.

Step Global Kimi shape Per GPU at TP=8 Communication?
qkv_a 7168 -> 2112 replicated full projection No TP communication for the projection itself.
q_b 1536 -> 64 x 192 1536 -> 8 x 192 No. Each rank computes its local heads.
KV cache c/k_nope[512] + k_rope[64] per token duplicated on each TP rank, then read by local heads No communication during the attention scan.
Scores [query, 64 heads, context] [query, 8 heads, context] No. Softmax is independent per head.
Value output 64 x 128 8 x 128 = 1024 No, still local.
o_proj 8192 -> 7168 1024 -> partial 7168 No inside Kimi attention. SGLang sets reduce_results=False, so this returns a partial hidden vector.
post-attention communicator sum TP partial hidden vectors partial 7168 -> full 7168 Yes. The all-reduce is deferred until after attention output projection.

Mental model: GPU0 does heads 0-7 all the way through score, softmax, and value. GPU1 does heads 8-15. This continues through GPU7. Each rank reads the duplicated latent cache for its local heads. Only after local head outputs are projected back to partial hidden space do ranks need to combine.

8. Why The Decode KV Scan Still Exists

MLA reduces cache size

Instead of reading full K and V for every head, Kimi reads the latent cache c/k_nope[512] and RoPE key k_rope[64]. This is much smaller.

But attention still scans context

For every generated token, each local head still needs one score per cached position. That is why decode remains KV-cache-sensitive.

Decode for one sequence with context length S: For each local head h: for j in 0..S-1: score_j = dot(q_content_h, c_j/k_nope_j) + dot(q_rope_h, k_rope_j) Then softmax over S positions and weighted sum over c_j/k_nope_j. MLA makes each j cheaper, but it does not remove the loop over j.
