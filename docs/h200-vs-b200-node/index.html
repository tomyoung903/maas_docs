Observed production nodes + published reference

8× H200 vs 8× B200 vs 8× B300

A three-way node comparison: passive inventory from one Europe H200 node and one Northwest B200 node, plus NVIDIA's published eight-GPU HGX/DGX B300 reference.

Observed 10 July 2026 H200 + B200 passively inspected B300 reference specifications only No benchmark or workload
The evidence is intentionally asymmetric. H200 and B200 describe machines that were passively inspected; no B300 cluster access path is documented in this environment, so B300 is an NVIDIA-published reference only. The H200 sample is also a virtualized guest while the B200 sample is bare metal. Each row identifies whether it is observed or published so these boundaries remain visible.

Takeaway

B300 is the capacity leader.

At 2.304 TB across eight GPUs, B300 offers 60% more marketed HBM than B200 and 104.3% more than H200.

B300 is not a blanket 2× B200.

Published FP8, HBM-bandwidth, and NVLink ceilings match B200. The clear compute change is dense FP4: 108 versus 72 PFLOPS, a 50% uplift.

Only H200 and B200 are observed here.

The B300 column is suitable for architectural planning, not for claims about installed drivers, actual power limits, NUMA locality, or production throughput.

Published Tensor Core peaks use NVIDIA's stated sparsity assumptions. Aggregate NVLink bandwidth sums per-GPU link bandwidth; it is not one transfer's bandwidth. NVIDIA's current Enterprise RA and DGX B300 guide state 8×288 GB = 2.3 TB, while the HGX product table separately lists 2.1 TB and one current trusted-computing SKU is named 270 GB. This page uses the explicit 288 GB reference but flags it as nominal; verify the ordered and driver-reported B300 SKU before capacity planning.

Topology and Network

All three designs keep eight GPUs inside one NVSwitch scale-up domain. CPU and NIC affinity were observed only on H200 and B200; the B300 diagram shows NVIDIA's published baseboard and networking reference without inventing a NUMA placement.

Serving Implications

Model and KV-cache capacity

The nominal B300 reference adds 864 GB over B200 and 1.176 TB over H200 per eight-GPU node. That is its clearest advantage for larger weights, KV pools, contexts, or concurrency.

Memory-bound inference

B200 and B300 both publish up to 64 TB/s per node. B300 can retain more state, but its top-line HBM bandwidth does not promise faster weight or KV streaming than B200.

Tensor-parallel collectives

B200 and B300 share a 14.4 TB/s aggregate NVLink ceiling, twice H200's. Do not expect B300 to improve TP collectives solely from the generation name.

Low-precision compute

B300 keeps B200's sparse FP8 and sparse FP4 peaks while raising dense FP4 from 72 to 108 PFLOPS. A model benefits only when its formats and kernels use that path.

Kernel and library target

B300 is compute capability 10.3, not B200's 10.0. CUDA libraries, custom kernels, containers, and build targets need explicit SM 10.3 support before production parity can be assumed.

Scale-out and facility planning

The B300 reference doubles the per-GPU network ceiling to 800 Gb/s and the DGX reference is a 14.5 kW system. Actual OEM wiring, power, cooling, and link state still require node inspection.

What this comparison does not prove: no claim is made about tokens per second, TTFT, inter-token latency, MFU, kernel efficiency, energy per token, or end-to-end GLM-5.2 speedup. In particular, NVIDIA's published “2× attention performance” for B300 is a platform-relative claim, not a result reproduced on these nodes.

Method and Sources

H200/B200 inspection

  • Kubernetes node readiness, product label, and advertised GPU count.
  • nvidia-smi --query-gpu for model, memory, power limit, compute capability, PCIe maximum, and driver.
  • nvidia-smi topo -m for GPU, NUMA, and NIC locality.
  • lscpu, /proc/meminfo, lspci, RDMA sysfs, and lsblk for host inventory.

B300 evidence

  • No B300 kubeconfig, SSH target, or cluster-access skill is documented in this environment.
  • Capacity and topology use NVIDIA's current HGX B300 Enterprise Reference Architecture.
  • Host, storage, power, and software examples use the NVIDIA DGX B300 user guide.
  • Compute and relative performance fields use NVIDIA's HGX product table and CUDA GPU list.

Explicitly not run

  • No CUDA context, PyTorch import, model request, synthetic kernel, or GPU benchmark.
  • No CPU benchmark, stress tool, compilation, package installation, or broad process scan.
  • No disk throughput, network throughput, collective, or RDMA traffic test.
  • No serving configuration changes, Kubernetes mutations, or guessed B300 login attempts.

Primary references

  • NVIDIA HGX AI Factory components: H200/B200/B300 HBM, B300 baseboard, host requirements, NVLink, and ConnectX-8 reference.
  • NVIDIA H200 Tensor Core GPU: H200 compute, memory, power, MIG, PCIe, and NVLink specifications.
  • NVIDIA DGX B200 SuperPOD components: eight-GPU FP8/FP4 and aggregate-memory specifications.
  • NVIDIA DGX B200 hardware overview: eight B200 GPUs, 1,440 GB total HBM, NVSwitch bandwidth, and ConnectX-7 networking context.
  • NVIDIA HGX platform specifications: B200/B300 FP8, sparse and dense FP4, total memory, NVLink, networking, and relative attention figures.
  • NVIDIA DGX B300 user guide: 8×288 GB memory, 72/144 PFLOPS, CPUs, system RAM, networking, storage, operating system, and 14.5 kW physical specification.
  • NVIDIA CUDA GPU compute-capability table: H200 9.0, B200 10.0, and B300 10.3.
  • NVIDIA Fabric Manager guide: B200/B300 baseboard topology with two NVSwitch ASICs and 18 NVLinks per GPU.
  • NVIDIA MIG supported-GPU table: current H200/B200 entries and the current absence of B300.
  • NVIDIA Trusted Computing release notes: current HGX B300 270 GB SKU naming, retained as a capacity-planning caveat.
