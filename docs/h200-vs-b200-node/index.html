8× H200 vs 8× B200 Production Nodes

A comparison of one Europe H200 serving node and one Northwest B200 serving node, combining passive machine inventory with NVIDIA-published GPU ceilings.

Observed 10 July 2026 One representative node per cluster No benchmark or workload
This is not a pure GPU-generation comparison. The observed H200 node is a virtualized guest with GPU passthrough, Virtio storage, and mlx5 virtual functions. The observed B200 node is bare metal with physical ConnectX controllers and local NVMe. GPU differences and platform-provisioning differences are separated below.

Takeaway

B200 raises the accelerator ceiling.

Per node: 27.7% more HBM, up to 66.7% more HBM bandwidth, 2× NVLink bandwidth, and roughly 2.27× NVIDIA-advertised sparse FP8 peak.

H200 has the lower GPU power envelope.

The observed limits total 5.6 kW for eight H200s versus 8.0 kW for eight B200s. B200's configured GPU ceiling is 42.9% higher.

Both nodes have strong scale-up and scale-out wiring.

Every GPU pair reports NV18, and each GPU has a nearby active 400 Gb/s NDR compute-fabric link.

Topology and Network

Both systems keep all eight GPUs inside one scale-up domain. Their CPU affinity differs: the H200 guest divides four GPUs per NUMA node; the B200 host divides two GPUs per NUMA node.

Serving Implications

Model and KV-cache capacity

B200 adds 312 GB of marketed HBM per node. At the same weight format and static-memory fraction, that can hold a larger KV pool, longer contexts, or more concurrency.

Memory-bound inference

The up-to-66.7% bandwidth uplift matters most when weight streaming, MoE expert reads, or attention/KV traffic is the limiting resource. It does not guarantee the same latency uplift.

Tensor-parallel collectives

The 2× NVLink ceiling gives B200 more headroom for all-reduce, all-gather, and all-to-all traffic inside the node. Real gains depend on message size, NCCL, topology use, and overlap.

Low-precision compute

B200's advertised sparse FP8 ceiling is about 2.27× higher and it adds FP4. GLM-5.2 benefits only when its kernels, quantization path, and numerical requirements use those units effectively.

Host-side scheduling

The B200 host has 50% more physical CPU cores and 28% more RAM, but four NUMA domains make CPU pinning, NIC affinity, and data placement more important.

Power and cooling

The observed B200 GPU limits allow 2.4 kW more accelerator power per node. Capacity planning must account for that higher ceiling even when average serving draw is lower.

What this comparison does not prove: no claim is made about tokens per second, TTFT, inter-token latency, MFU, kernel efficiency, energy per token, or end-to-end GLM-5.2 speedup. Those require controlled benchmarks on otherwise aligned software and serving configurations.

Method and Sources

Passive node inspection

  • Kubernetes node readiness, product label, and advertised GPU count.
  • nvidia-smi --query-gpu for model, memory, power limit, compute capability, PCIe maximum, and driver.
  • nvidia-smi topo -m for GPU, NUMA, and NIC locality.
  • lscpu, /proc/meminfo, lspci, RDMA sysfs, and lsblk for host inventory.

Explicitly not run

  • No CUDA context, PyTorch import, model request, synthetic kernel, or GPU benchmark.
  • No CPU benchmark, stress tool, compilation, package installation, or broad process scan.
  • No disk throughput, network throughput, collective, or RDMA traffic test.
  • No serving configuration changes and no Kubernetes mutations.
