CUDA execution foundations

CUDA Kernel Fundamentals on H200

A practical model of how PyTorch and SGLang work becomes CUDA streams, kernel launches, thread blocks, warps, SM occupancy, and per-block shared-memory allocations on an NVIDIA H200.

Audience: inference engineers Hardware: Hopper / H200 Updated: July 16, 2026

One Mental Model

The complete relationship: a stream orders kernel launches; one kernel launch creates a grid of thread blocks; CUDA assigns those blocks to SMs; each block executes as warps and receives a private shared-memory allocation from its resident SM.
Ordering CUDA stream

A software command queue. Operations in one stream observe stream ordering.

Invocation Kernel launch

One invocation of GPU code, represented as a grid containing many blocks.

Work unit Thread block

A cooperating group of threads that must remain on one SM.

Hardware SM

A physical execution engine that schedules warps from resident blocks.

On-chip storage Shared memory

Fast scratch space allocated per block from the SM where that block resides.

Supporting hierarchy: kernel grid → thread blocks → 32-thread warps → individual threads.

The Five Concepts

Stream

A CUDA stream is an ordered queue of commands such as kernel launches, memory copies, event records, and waits. It is a CUDA concept; PyTorch wraps it with torch.cuda.Stream.

Owns
Ordering and dependency relationships.
Does not own
An SM, fixed cores, or a physical execution lane.
Profiler
Usually appears as a lane containing ordered operations.

Kernel

A kernel is a GPU function invocation. Its launch describes a grid of blocks, the block shape, dynamic shared-memory bytes, and the destination stream.

Owns
A grid containing zero or more waves of blocks.
Does not mean
One thread, one block, or one SM.
Profiler
One event spanning that invocation's GPU execution interval.

Thread Block

A block is a group of threads that can synchronize and exchange data through shared memory. CUDA places the complete block on one SM for its lifetime.

Contains
Threads grouped by hardware into 32-thread warps.
Can
Share data and synchronize internally.
Cannot
Split across SMs or depend on arbitrary block execution order.

Streaming Multiprocessor

An SM is a physical processor inside the H200. It contains warp schedulers, Tensor Cores, arithmetic pipelines, load/store units, registers, and the unified L1/shared-memory resource.

Hosts
One or more resident blocks when resources permit.
Executes
Ready warps selected from those resident blocks.
Reuses
Freed capacity as blocks complete and new blocks arrive.

Shared Memory

Shared memory is programmable, low-latency on-chip storage. The storage physically belongs to an SM, but CUDA allocates a private region to each resident block. That region disappears when the block completes.

Scope
Threads in one block; Hopper clusters provide an explicit distributed-shared-memory extension.
Purpose
Reuse tiles, exchange values, stage HBM data, and reduce global-memory traffic.
Not
The H200's 141 GB HBM3e model and KV-cache storage.

From SGLang To Hardware

SGLang decides which stream to use, PyTorch exposes the stream API and launches operators, and CUDA implements the queue and schedules the resulting blocks on hardware.

# SGLang asks PyTorch for a CUDA stream.
forward_stream = torch.cuda.Stream()

# This context makes it the current stream.
with torch.cuda.stream(forward_stream):
    output = model(inputs)

Operators called inside the context normally launch their CUDA kernels into the current stream. Libraries may also use coordinated auxiliary streams.

  1. 1
    SGLang chooses the policy.

    For example, one dominant forward stream per PP rank plus scheduler, copy, and communication streams.

  2. 2
    PyTorch creates a wrapper.

    torch.cuda.Stream() requests an underlying CUDA stream from the runtime.

  3. 3
    An operator selects a kernel.

    Shape, dtype, architecture, and backend determine the GEMM, attention, routing, or elementwise variant.

  4. 4
    CUDA enqueues the launch.

    The launch carries grid dimensions, block dimensions, shared-memory requirements, and the stream.

  5. 5
    Hardware schedules blocks.

    Available SMs accept blocks when dependencies are satisfied and sufficient registers, warps, and shared memory remain.

A stream is not hardware. It has no permanent mapping to an SM. Successive kernels from one stream may use every SM, a subset of SMs, or wait without executing because an earlier dependency has not completed.

What actually executes?

  • A block's threads are divided into warps of 32 threads.
  • Warp schedulers select ready warps from all resident blocks.
  • Tensor Cores and arithmetic pipelines execute instructions issued by those warps.
  • A stalled warp does not necessarily stall the SM; another ready warp may issue.
  • A single active kernel can therefore contain thousands of simultaneous threads across many SMs.

Shared Memory Is Not HBM

H200's global HBM and an SM's shared memory serve different roles. The difference explains why a kernel can fail its shared-memory launch shape even while tens of gigabytes of HBM remain free.

Lifetime: when CUDA places a block on an SM, it reserves that block's registers and shared memory. The allocation remains resident until the block finishes, then becomes available to another block. It is not a persistent cache of request data.

The measured low-shared-memory fallback

Srequested = 327,684 bytes ≈ 320 KiB > 232,448 bytes = 227 KiB

The 16,384-token kernel variant requested more shared memory for one block than Hopper permits. Selecting a low-shared-memory variant was therefore a kernel-configuration fallback, not a 141 GB HBM out-of-memory condition.

Residency And Occupancy

An SM can host multiple blocks concurrently only when all resource constraints fit. Shared memory is one constraint; registers, threads, warps, block limits, barriers, and cluster rules also matter.

Nresident ≤ min(32, ⌊64 / Wblock⌋, ⌊65,536 / Rblock⌋, ⌊228 KiB / Sblock⌋)

Wblock is warps per block, Rblock is total 32-bit registers allocated to the block, and Sblock is shared memory per block. This is a first-order upper bound; allocation granularity and other architectural constraints can lower the actual result.

64 KiB per block

Shared memory alone permits at most three resident blocks per SM: ⌊228 / 64⌋ = 3.

200 KiB per block

Shared memory alone permits only one resident block per SM. The kernel can run, but co-residency is restricted.

More than 227 KiB

One block exceeds Hopper's addressable per-block limit. That launch shape cannot run and needs another kernel variant.

Resident

Hardware state and resources are allocated. A resident warp or NCCL kernel may still be waiting rather than issuing useful instructions.

Occupancy

Active resident warps relative to the architectural maximum. It is a latency-hiding capacity metric, not direct useful-work utilization.

Utilization

How busy a hardware resource is over time. Tensor Core, memory-bandwidth, and SM-active utilization answer different questions.

How To Read A Kernel Timeline

A profiler rectangle is an execution interval for one kernel launch on one GPU. Its width alone does not reveal how many SMs were active, how many blocks were resident, or whether every resident warp was doing useful work.

What you seeWhat it meansWhat it does not prove
One long GEMM rectangleOne kernel invocation remained active for that interval.It does not mean one block or one SM was working.
Back-to-back kernels on one streamThe stream and data dependencies serialized those launches.It does not mean the H200 lacks concurrent-kernel support.
Long NCCL SendRecv rectangleThe communication kernel was resident for that interval.It does not mean payload bytes moved usefully for the entire interval.
White gap with no GPU kernelNo retained GPU kernel event covered the interval.It does not prove the process was idle; CPU scheduling, JIT compilation, synchronization, or uncollected activity may be present.
Tiny apparent overlapProfiler timestamps overlap by a small interval.Microsecond boundary effects do not necessarily establish useful concurrent execution.
Ask a second question after duration: how many blocks and warps were active, which SM resources were limiting residency, and whether the kernel was computing, transferring, or waiting?

  • The eight ranks did not share one CUDA stream; stream identifiers are local to each process.
  • Within a rank, dependent model kernels ran almost entirely on its dominant forward stream.
  • Across ranks, a downstream NCCL receive could remain resident while the upstream GPU computed.
  • With one request and one microbatch, useful model compute advanced mostly stage by stage instead of filling the pipeline.

Kernel Analysis Checklist

Use this sequence when a timeline looks surprising:

  • Which process, GPU, and CUDA stream launched the kernel?
  • What dependency prevents the next operation from starting?
  • How many blocks are in the grid, and how many threads are in each block?
  • How many warps, registers, and shared-memory bytes does one block consume?
  • How many blocks can reside on one SM under those constraints?
  • Does the grid contain enough blocks to occupy the available SMs?
  • Is the kernel compute-bound, memory-bound, communication-bound, or waiting?
  • Does a long duration represent useful work or merely residency?
  • Could another stream overlap, or are data dependencies decisive?
  • Was the exact shape warmed up before profiling, including compiled paths?
