Kubernetes discovery

A worker is routable only after Kubernetes readiness and Dynamo metadata agree.

A decode pod can be running before it is safe for the frontend to send it new requests. Dynamo avoids that by joining two signals: Kubernetes says the pod is ready, and Dynamo says the pod has registered a usable generate endpoint with model metadata.

Operator/deployment Decode worker Kubernetes readiness Dynamo discovery Frontend/router
DynamoWorkerMetadata CR The worker publishes what it can do: namespace, component, endpoint, instance id, transport, and model/runtime metadata.
EndpointSlice readiness Kubernetes publishes which selected pods are ready according to their readiness probes.
Frontend worker set The frontend watches the joined result and routes only to workers that have both signals.
pod starts runtime starts handler installed metadata CR written /health ready EndpointSlice ready discovery joins frontend routes
Code-order nuance. In the actual runtime code, serve_endpoint installs the request handler and marks endpoint health ready before the discovery registration persists metadata into Kubernetes. The externally visible rule remains the same: the frontend sees the worker only after the discovery daemon can join readiness and metadata.

10-Step Flow With Code Anchors

  1. Decode pod starts

    deployment

    The chart values describe a Dynamo graph with a frontend service and a worker service. The worker defaults to the service name decode; the GLM CP overlay sets image, model path, replica count, resources, probes, and SGLang flags. The Dynamo operator turns that high-level graph into lower-level Kubernetes objects.

    Why this mattersAt this point Kubernetes can create a pod, but the model process may still be loading and not yet useful.

  3. generate endpoint handler is installed

    worker process

    Why this mattersDiscovery metadata alone is not enough; the actual request handler must be installed first.

  4. Runtime writes DynamoWorkerMetadata CR

    discovery metadata

    Dynamo's discovery API is an address book abstraction. With the Kubernetes backend, registering an endpoint means writing a namespaced DynamoWorkerMetadata custom resource. That CR stores serialized discovery metadata, and it has a pod owner reference so Kubernetes can garbage-collect it when the pod disappears.

    Why this mattersThe frontend needs model/runtime metadata, not just a pod IP.
    Plain languageThis is the worker saying "I am worker X, I provide generate, and here is how to talk to me".

  5. /health becomes ready after required endpoint health is ready

    system health

    Worker pods expose a system server on port 9090. The operator configures system health to depend on the generate endpoint. Once the handler is inserted, the runtime marks that endpoint ready, and /health can return healthy.

    Why this mattersReadiness should reflect the endpoint that will actually serve generation requests.
    Plain languageKubernetes asks "are you ready?", and the worker answers yes only after generate is usable.

  6. kubelet readiness probe succeeds

    pod readiness

    The kubelet is the node-local Kubernetes agent. It periodically calls the worker's readiness probe. In your GLM CP values, readiness is GET /health on the named system port, with a long initial delay and a one-failure threshold.

    Why this mattersKubernetes will not mark the pod as ready until the probe succeeds.

  7. Kubernetes EndpointSlice marks pod ready

    cluster readiness

    The operator creates a Service with discovery labels and selectors. Kubernetes maintains EndpointSlices for that Service. Each EndpointSlice entry records whether the selected pod is ready. Dynamo reads those ready entries as the Kubernetes side of the gate.

    Why this mattersEndpointSlice is the cluster-wide readiness signal that other pods can watch.
    Plain languageKubernetes publishes a list saying "these selected pods are ready right now".

  8. Dynamo discovery daemon joins readiness and metadata

    join gate

    Every pod using Kubernetes discovery runs a discovery daemon. In pod mode it watches EndpointSlices for readiness and watches DynamoWorkerMetadata CRs for metadata. A worker enters the snapshot only when the ready EndpointSlice entry has a matching metadata CR.

    Why this mattersThis prevents both early routing and stale routing.
    Plain languageThe daemon says "I need both the green light and the worker's business card".

    • dynamo/lib/runtime/src/discovery/kube/daemon.rs:278 iterates ready entries

  9. Frontend discovery watch receives the worker

    watch stream

    The frontend starts a DistributedRuntime with the configured discovery backend. Its HTTP entrypoint asks discovery to list_and_watch all model cards. The model watcher consumes discovery events and updates the in-memory model/worker manager.

    Why this mattersThe frontend does not poll random pod IPs; it follows the Dynamo discovery stream.
    Plain languageThe frontend subscribes to the address book and notices when a worker joins or leaves.

    • dynamo/lib/llm/src/discovery/watcher.rs:199 handles added model instances

  10. Frontend/router can send new requests to it

    routing set

    Routing uses a worker set built from live instances and runtime configs. The KV router chooses a worker; it does not make the worker exist. Discovery decides which workers are eligible, and routing chooses among those eligible workers for each request.

    Why this mattersA worker must be both live and configured before the router should select it.

Glossary For Kubernetes/Dynamo Laymen

Operator
A Kubernetes controller that watches high-level Dynamo custom resources and creates lower-level Kubernetes objects for you. You write a DynamoGraphDeployment; the operator reconciles component deployments, services, probes, labels, env vars, service accounts, roles, and role bindings.
DynamoGraphDeployment
The high-level graph object that describes the app: frontend service, decode worker service, replica counts, image, args, resources, probes, and volumes. In your GLM chart this is rendered by Helm from glm-5.2.cp.yaml.
DynamoComponentDeployment
The operator's intermediate component object. The graph has several services; each service becomes a component deployment, and then the component deployment controller creates or updates the real Kubernetes Deployment.
Decode pod / backend / worker
In this page these are the same traffic-serving side of the system. The chart calls the worker service decode. The runtime endpoint is usually discussed as the backend generate endpoint because it is the thing the frontend eventually calls.
Frontend
The HTTP-facing process. It receives OpenAI-compatible requests, watches discovery, maintains a current view of available model workers, and asks a router to choose a worker for each request.
Discovery
Dynamo's address-book abstraction. It answers "which components exist, what endpoints do they provide, what model/runtime metadata do they have, and are they currently eligible?" On Kubernetes, this address book is backed by native Kubernetes objects.
DWM: DynamoWorkerMetadata
The actual custom resource used by Kubernetes discovery. It stores serialized Dynamo metadata for a worker pod. DWM is the shorthand used in this page; DynamoWorkerMetadata is the exact code and Kubernetes resource term.
Custom Resource and CRD
A custom resource is a Kubernetes object type added by an application. The CRD defines the type; each CR is one object instance. DynamoWorkerMetadata is a custom resource under the nvidia.com API group.
EndpointSlice
A Kubernetes object maintained for Services. It lists selected pod endpoints and their readiness condition. Dynamo watches EndpointSlices so it can use Kubernetes' view of readiness without directly probing every pod itself.
readinessProbe
A kubelet check that decides whether a pod should be considered ready. For this worker it calls /health on the system port. A passing readiness probe allows Kubernetes to mark the endpoint ready.
livenessProbe and startupProbe
Liveness asks whether Kubernetes should restart a stuck container. Startup gives slow-starting containers time before liveness restarts apply. They protect process lifecycle, while readiness controls whether the pod should receive new traffic.
kubelet
The Kubernetes node agent. It runs containers through the container runtime and performs probes. When its readiness probe succeeds or fails, that status flows back into Kubernetes pod and EndpointSlice state.
Request plane
The transport and server path that carries actual generation calls. Installing an endpoint in the request plane means the worker has a callable handler, not just metadata saying it exists.
Router / KV router
The routing logic in the frontend. Discovery determines which workers are candidates. The router chooses one candidate using policy and runtime information such as KV/cache-aware state.
Owner reference
A Kubernetes link saying one object belongs to another. DWM CRs are written with a pod owner reference, so Kubernetes can clean up metadata when the pod is deleted.
RBAC
Kubernetes permissions. The operator creates a service account, role, and role binding so Dynamo pods can watch EndpointSlices and DWM CRs in their namespace.

Why The Join Matters

The most important line is not a single probe or a single CR write. It is the join: ready EndpointSlice entry + matching DynamoWorkerMetadata CR. Each signal blocks a different failure mode.

EndpointSlice ready? DWM CR exists? Published to frontend? Meaning
No No No The pod is absent, starting, or not useful yet. Nothing should route to it.
No Yes No The worker has metadata, but Kubernetes does not consider it ready. Do not route yet.
Yes No No The pod looks ready, but Dynamo does not have endpoint/model metadata. Do not route yet.
Yes Yes Yes The pod is ready and Dynamo knows what it provides. The frontend can include it.
